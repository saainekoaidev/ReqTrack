# 0018 レビュー工程の有無選択とレビュワー/レビュイーの計上

## Status

Accepted (US-014 のレビュー自動展開を拡張)

## Context

レビュー工程は案件により要否が分かれる。また従来は機能(level1)単位で 1 本のレビュー(PL割付)のみを生成しており、(1)生成時に有無を選べない、(2)3 階層(対象=画面/帳票)に対応していない、(3)レビュワーだけで「レビュイー側の工数」が載らない、という不足があった。

## Decision

- **生成時にレビュー工程の有無を選択**: ガント作成(新規作成の「この見積でガントを作成」)および進捗管理の「スケジュール再生成」に「レビュー工程を入れる」チェックを設ける。
  - エンドポイント `POST /api/projects/:id/reviews { include }`: include=true で展開、false で `kind='review'` を全削除。生成前に呼ぶ。
- **対象(level2)単位**で展開(無ければ機能 level1)。各工程ルール(基本設計/詳細設計/コーディング/単体テスト)について:
  - **レビュワー側**タスク(`(レビュー)`)を生成し、**最上位(プロジェクト管理者)→チームリーダー**へ自動割付(複数レビュワーでも、いったん全て最上位へ。後で WBS編集で振り分け→再生成)。
  - **レビュイー側**タスク(`(レビュー対応)`)を、その工程の開発担当(代表として最初の担当)へ生成(担当が居て、レビュワーと別人の場合)。
- レビュー工数は設定のレビュー率/下限(`computeReviewTasks`)に従う。

## Consequences

- レビュー要否を案件ごとに選べ、不要なら一切登場しない。
- レビュワー/レビュイー双方の工数が計上され、要員別工数・工賃にも反映される。
- 振り分け(PM→TL)は WBS編集での担当変更＋再スケジュールで行う(スケジュール見直しを伴う)。
- 限界: レビュワー review とレビュイー review は同一対象(親)配下のため、現スケジューラでは厳密な同時刻ではなく前後に直列化されうる(同時実施の正確なモデル化は将来課題。明示依存リンク [0015] と同様)。
