# 0015 要員・依存を考慮したスケジューリングと実行中の WBS 編集

## Status

Accepted (US-004 / US-040 の「全タスクを作成順に 1 本で直列化」を置き換える)

## Context

従来のスケジューラは全ての葉タスクを作成順に 1 本の鎖で直列化していた。そのため、(1)誰がどのタスクをやるか(要員)が日程に反映されず、(2)タスク同士のつながり(依存)も無いため、「要員を変えるとどこを起点に日程が見直されるのか」が定義されていなかった。また、見積後(実行フェーズ)に 工数・稼働率・担当を変えたり、タスクを追加/削除したりする場所が無かった(編集は新規作成フローのみ)。

## Decision

### スケジューリングモデル(再生成の規則)

開始日を起点に、次の 2 つの制約で全タスクを流し直す(`scheduleTasks`):

- **依存(工程順)**: 同一の対象(level2 = 同じ親)配下の作業は、WBS 番号順(基本設計→詳細設計→…→結合テスト)に**直列**(finish-to-start)。対象・機能をまたぐタスクは互いに**並行**可能。
- **要員(リソース)**: 同一要員のタスクは**直列**(1 人が同時に複数を実施しない)。未割当タスクは共通プール `(未割当)` として直列。異なる要員は**並行**。

各タスクの開始 = max(プロジェクト開始, 同一対象の直前作業の終了, 同一要員の直前タスクの終了)。これにより:
- 要員を**追加**(分担)すると並行化で全体が短縮、
- 要員を**変更**するとその要員の鎖が組み替わり、
- **工数/稼働率**を変えると以降が後ろ/前倒し、
と影響が予測可能になる。**明示的なタスク間リンク**(機能をまたぐ依存、例: 機能Bのコーディングが機能Aの結合を待つ)は本 ADR では持たない(将来課題)。

### 編集のライフサイクル(どこで何をするか)

- **新規作成**: 見積確認(工数/稼働率/根拠/担当, US-039)、または手組み WBS。
- **実行フェーズ**: 「進捗管理 > **WBS編集**」で タスクの追加/削除/変更、工数・稼働率・担当の変更を行い、「**スケジュール再生成**」で日程へ反映。要員の追加/削減は「設定 > 要員」、割付変更は WBS編集で行う。
- WBS 編集テーブルは新規作成と進捗管理で同一コンポーネント(`WbsEditor`)を共有する。

## Consequences

- 要員の増減・付け替えがそのまま日程に反映され、再生成の起点(開始日)と直列条件(対象内工程順・要員別)が明確。
- 実行中の計画変更(タスク追加/工数調整/担当替え)が 1 か所で完結する。
- 明示的な依存リンクが無いため、機能をまたぐ厳密な順序は表現できない(必要なら後続 ADR で predecessor リンクを追加)。
