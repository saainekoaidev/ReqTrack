# 0013 WBS 3 階層(機能→対象→作業)と AI 見積の構造

## Status

Accepted (US-013 の 3 階層 WBS を AI 見積にも適用)

## Context

AI 見積(US-036)は当初 機能(level1)→作業(level3) の 2 階層で生成しており、対象(level2)が無いため WBS が 1.1 止まりで、機能(1.)直下にいきなり作業が並んでいた。実務では「どの画面/帳票に対する作業か」(対象)が抜けると、見積の内訳が追えず、ガントのインデント 3(1.1.1)も成立しない。

## Decision

AI 見積を **機能(feature) → 対象(target) → 作業(task)** の 3 階層で生成する。

- **対象(target)** は機能が扱う「画面」または「帳票」とし、name は `画面ID(画面名)` / `帳票ID(帳票名)` の形式(例: `SCR-001(ログイン画面)`, `RPT-010(売上日報)`)。この対象の存在がインデント 3(1.1.1)掲出のキーになる。
- 作業(task)は対象ごとに標準工程(基本設計/詳細設計/コーディング/単体テスト/結合テスト)へ分解する。
- 出力 JSON は `{features:[{name, targets:[{name, tasks:[{name,phase,estimateDays,reason}]}]}]}`。
- 後方互換: 旧形式(feature 直下に tasks)を受けたら「<機能名> 全般」という単一の対象でくるんで正規化する。
- 永続化は level1(機能, wbsId `1`) / level2(対象, `1.1`) / level3(作業, `1.1.1`)。

## Consequences

- 見積内訳が「機能→画面/帳票→作業」で追え、ガントの 3 階層ツリー(US-040)が成立する。
- 手組み WBS(WbsEditPage)は従来どおり level1〜3 を任意に組める(対象を level2 として作成可能)。
- ガントの行表示・折り畳み・小数日対応の描画は US-040(別 ADR)で扱う。
