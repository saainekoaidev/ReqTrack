# 0016 進捗のあるタスクを固定する再スケジュール

## Status

Accepted (US-041 の再スケジュールを拡張)

## Context

US-041 の再スケジュールは進捗(実績)を見ずに全タスクを開始日から引き直していた。そのため、完了済み・着手中のタスクまで未来へ移動してしまい、要員変更・前段タスク挿入・工数変更を行うと実績と計画が食い違う恐れがあった。「進捗が入っているタスクに変更が入ったらどうなるか」を定義する必要がある。

## Decision

再スケジュール(`scheduleTasks`)は進捗で挙動を分ける:

- **進捗 100%(完了)**: `plannedStart`/`plannedEnd` を**固定**(動かさない)。
- **進捗 1〜99%(着手中)**: **開始を固定**し、工数(人日)・稼働率の変更に応じて**終了のみ伸縮**(固定開始から所要稼働時間を割り付け直す)。担当変更・前段挿入では動かさない。
- **進捗 0%(未着手)**: 従来どおり 開始日・依存(対象内工程順)・要員 で引き直す。

着手済み/完了タスクは「アンカー」として後続の起点(同一対象・同一要員の直前終了)に寄与するが、鎖の都合で**過去へ巻き戻さない**(`groupLastEnd`/`resourceLastEnd` は最遅終了を採用)。前段に未着手タスクを挿入しても、着手中タスクは後ろへ押し戻さず、未着手側が周囲に流れて調整される。

判定には各タスクの `progress` と既存 `plannedStart/End`(アンカー)を `scheduleProject` から渡す。

## Consequences

- 実績(着手・完了)が再計画で壊れない。要員/工数/担当の変更や前段挿入の影響は未着手タスクに限定され予測可能。
- 着手中タスクの工数を増やすと終了が伸び、後続(未着手)がその後に流れる。
- 着手中タスクの「開始の手動修正」は別途(WBS編集や将来の手動ドラッグ)で行う前提(本 ADR では自動では動かさない)。機能をまたぐ明示依存が無い点は [0015](0015-resource-dependency-scheduling.md) のとおり将来課題。
