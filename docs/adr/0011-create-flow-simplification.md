# 0011 新規作成フローの簡素化(3ステップ + 作成中プロジェクトのロック)

## Status

Accepted (US-008 のシェル / US-019 取込 / US-023 ステッパーの一部を見直す)

## Context

実運用で新規作成フローが分かりにくいという指摘:

- ステッパーのステップ1(プロジェクト作成)へいつでも戻れ、戻ると名称入力が再びでき別プロジェクトを作り直せてしまう。
- 「要件一覧ファイル」と「見積明細ファイル」の 2 つの取込枠があるが、利用者の実態では取り込むファイルは「要件一覧(=見積明細)」1 つ。中身が顧客要件のみか、弊社改修案まで含むかの違いだけ。
- 取込時に標準工程 WBS を先行展開していたため、AI 見積実行後に工数0の WBS が残る。WBS を見積前に持っている必要はない。
- 「次へ」の導線がリンク主体で分かりにくい。

## Decision

新規作成フローを 3 ステップに簡素化する:

1. **プロジェクト作成** (`/create`): 名称・概要・案件区分(新規/既存)・参照資料を入力し作成。作成後はステップ1の入力を**非活性表示**にし、操作は「やり直し」(作成中プロジェクトを破棄して最初から)と「次へ:要件登録」のみ。作成中プロジェクトは `CreateContext`(localStorage `reqtrack.createProjectId`)で全ステップ共有する。
2. **要件登録** (`/create/requirements`): **「要件一覧(見積明細)ファイル」1 つ**の取込、または自然文で要件を登録。見積明細の二重アップロードは廃止。**取込時に WBS は展開しない**(`expand` 既定 false)。
3. **見積・ガント生成** (`/create/estimate`, 手組みは `/create/wbs`): ここで「AIで要件からガントまで生成」「AIで見積だけ生成」「手でWBSを組む(ガントから)」を選ぶ。**WBS(タスク)はこの選択後に生成**する。レビュー自動展開・効率化調整もこのステップに置く。ガント生成成功で作成中プロジェクトを確定(draft クリア)し進捗管理へ遷移。

各ステップは明確な「次へ」ボタンで進む。「見積から/ガントから」の起点選択はステップ1からステップ3へ移動した。

## Consequences

- フローが直線的になり、戻っても作り直しではなくロック表示+やり直しに限定され、誤って別プロジェクトを量産しない。
- 取込が 1 ファイルに統一され「見積明細とは何か」という混乱が無くなる。WBS は見積/ガントの起点選択後にのみ生成され、工数0の残骸が出ない。
- 旧ルート `/create/import`・`/create/tasks` は廃止(それぞれ requirements・estimate へリダイレクト)。`TasksPage`/`ImportPage` は撤去。見積明細(構造化タスク)直接取込のバックエンドは残すが UI からは外す。
- バックエンド `import/*` の `expand` 既定を false に変更(明示 true 時のみ展開)。
