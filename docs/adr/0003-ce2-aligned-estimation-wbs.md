# 0003 見積・WBS・スケジュールの ce2 準拠

## Status

Accepted

## Context

既存ワークスペース `C:\_bis\ce2`(GSSZ 改修の WBS/スケジュール生成資産)で、要件一覧→工数算出→ガント表→Excel(.xlsx, mpp 代替)出力の運用が確立している。ユーザより「見積の考え方・タスクの考え方・ガントの列項目を ce2 に強く合わせたい」「見積諸元と見積根拠を Excel に示してエクスポートしたい」との要望があり、フル準拠で取り込む方針となった([0002](0002-stack-and-architecture.md) の初期見積モデルを拡張する)。

ce2 の確立済み概念(ce2 `_adr/` 由来):
- 見積は「**正式**」一本(概算との二本立ては漏れ工程で逆転するため廃止 = ce2 ADR-0003)。
- 最小単位 **0.25 人日 = 1 コマ**、1 日 = 4 コマ(08-10 / 10-12 / 13-15 / 15-17)。人日→コマ = `round(人日 × 4)`。
- **工数推定理由(= 見積根拠)** をタスク行ごとに必須で持つ。
- **効率化調整**: 複数機能同時実施の重複削減を「負の人日」1 行 + 該当工程への注釈で表現し、合計を顧客見積と一致させる(ce2 ADR-0005)。
- **レビュー自動展開**: `review_rules`(after 工程 / ratio 0.3 / min 0.25 / reviewer=PL)で開発タスク後に PL レビューを自動挿入。dev のレビュー対応は明示しない。
- タスクは **機能(親)→ 対象(画面/帳票/テーブル)→ 作業タスク** の 3 階層、wbs_id = `1 / 1.1 / 1.1.1`、レビューは `2r設` 形式。
- 標準工程: 基本設計 / 詳細設計 / コーディング / 単体テスト / 結合テスト(+ 各レビュー)。
- メンバーは `role`(PL/developer)+ `assign`(担当機能)。タスク担当未指定は機能担当を継承。
- 土日・祝日を除いた営業日で、担当者ごとに直列(同日コマ継続)割付。

## Decision

ReqTrack のデータモデルとロジックを ce2 準拠に拡張する。

### データモデル(Prisma)
- `Task` に追加: `phase`(工程), `estimateNote`(見積根拠), `kind`(`task` | `review` | `efficiency`), `parentId`(自己参照) / `level`(1-3) / `wbsId`, `plannedStartQ` / `plannedEndQ`(コマ 0-3)。`estimateDays` は 0.25 刻みを許容(効率化調整は負値)。
- `Member` に `role`(既存)を `PL` | `developer` 等として運用。機能担当割当はタスク生成時に継承。

### 見積・スケジュール
- 見積単位は 0.25 人日。スケジューラはコマ(4/日)粒度で、営業日(土日祝除外)を担当者ごとに直列割付し、同日コマ継続を行う。
- レビュー自動展開・効率化調整・標準工程展開を純粋関数(`backend/src/domain/`)に実装し、ユニットテストで担保する。
- 効率化調整(負の工数)はガントのバー描画対象外とし、合計と Excel/根拠側で表現する(MS Project XML は将来課題)。

### 工程の標準セット
`基本設計 / 詳細設計 / コーディング / 単体テスト / 結合テスト` を既定とし、各開発工程の後に対応するレビュー工程(基本設計レビュー / 詳細設計レビュー / コードレビュー / 単体テストケースレビュー)を自動挿入する。

## Consequences

- ce2 のスケジュール表・見積根拠と同じ見方が ReqTrack 上で再現でき、出力(US-016 Excel)も顧客提示に耐える。
- 既存の日数 ceil ベースのスケジューラ(0002/US-004)はコマ粒度へ置き換える。整数人日では従来結果と一致するため後方互換は概ね保たれる。
- データモデルが複雑化(自己参照階層・工程・kind)するため、生成ロジックは純粋関数に隔離してテスト容易性を確保する。
- 概算モードは持たない(正式一本)。将来必要になれば本 ADR を Superseded で更新する。
