# 0003 見積・WBS・スケジュールの ce2 準拠

## Status

Accepted (2026-06-07 改訂: 0.25/コマ粒度を廃止し稼働率モデルへ。工程/見積根拠は見積粒度から分離)

## Context

既存ワークスペース `C:\_bis\ce2`(GSSZ 改修の WBS/スケジュール生成資産)で、要件一覧→工数算出→ガント表→Excel(.xlsx, mpp 代替)出力の運用が確立している。ユーザより「見積の考え方・タスクの考え方・ガントの列項目を ce2 に強く合わせたい」「見積諸元と見積根拠を Excel に示してエクスポートしたい」との要望があり、フル準拠で取り込む方針となった([0002](0002-stack-and-architecture.md) の初期見積モデルを拡張する)。

ce2 の確立済み概念から取り込むもの:
- 見積は「**正式**」一本(概算との二本立ては漏れ工程で逆転するため廃止 = ce2 ADR-0003)。
- **工数推定理由(= 見積根拠)** をタスク行ごとに持つ。
- **効率化調整**: 複数機能同時実施の重複削減を「負の人日」1 行 + 該当工程への注釈で表現し、合計を顧客見積と一致させる(ce2 ADR-0005)。
- **レビュー自動展開**: 開発タスク後に PL レビューを自動挿入(ratio/min/reviewer)。dev のレビュー対応は明示しない。
- タスクは **機能(親)→ 対象(画面/帳票/テーブル)→ 作業タスク** の 3 階層、wbsId = `1 / 1.1 / 1.1.1`、レビューは `2r設` 形式。
- 標準工程: 基本設計 / 詳細設計 / コーディング / 単体テスト / 結合テスト(+ 各レビュー)。
- メンバーは `role`(PL/developer)+ `assign`(担当機能)。タスク担当未指定は機能担当を継承。
- 土日・祝日を除いた営業日で、担当者ごとに直列割付。

### ce2 から取り込まないもの(2026-06-07 ユーザ指示で改訂)
- **0.25 人日粒度 / コマ(4 コマ/日)の概念は採用しない**。これは ce2 が Excel のセル割(セルグリッド)に合わせた都合であり、ReqTrack では不要。
- 代わりに **工数は小数点第 3 位までの自由値** とし、**稼働率(utilization)** の概念を導入する。

## Decision

### 見積の単位と稼働率(US-012)
- `Task.estimateDays`(工数, 人日)は **小数点第 3 位まで** の自由値(効率化調整は負値)。0.25 刻みやコマは持たない。
- `Task.utilizationRate`(稼働率, 0 < r ≤ 1, 既定 1.0)を導入。**工数(実作業量)とカレンダー期間の関係**を次式で表す:
  - `工数(人日) = 期間(営業日) × 稼働率`
  - スケジュール上の **期間(営業日) = 工数 ÷ 稼働率**(切り上げて営業日に割付)
  - 例: 3 営業日スパン × 100% = 3 人日(24h)、× 20% = 0.6 人日(4.8h)。逆に 0.6 人日 ÷ 20% = 3 営業日。
- 工数は人日が正本。時間表示が必要な箇所は `人日 × 8h` で換算する(8h/日)。

### 工程・見積根拠(US-013 側で扱う / 見積粒度とは分離)
- `Task.phase`(工程区分)と `Task.estimateNote`(見積根拠 = 工数推定理由)を持つ。
- これらは「稼働率による工数算出(US-012)」とは別の関心事として WBS/標準工程展開(US-013)で付与する。

### データモデル(Prisma)
- `Task` に追加: `utilizationRate`(Float 既定 1.0), `phase`(String?), `estimateNote`(String?), `kind`(`task` | `review` | `efficiency`), `parentId`(自己参照) / `level`(Int 1-3) / `wbsId`(String?)。`estimateDays` は自由小数・負値許容。
- `Member.role`(既存)を `PL` | `developer` 等として運用。機能担当割当はタスク生成時に継承。

### スケジュール
- スケジューラは **営業日単位**(土日祝除外)で、各タスクの期間 = `ceil(工数 ÷ 稼働率)` 営業日として担当者ごとに直列割付する(コマ分割はしない)。
- レビュー自動展開・効率化調整・標準工程展開・期間算出を純粋関数(`backend/src/domain/`)に実装し、ユニットテストで担保する。
- 効率化調整(負の工数)はガントのバー描画対象外とし、合計と Excel/根拠側で表現する(MS Project XML は将来課題)。

### 工程の標準セット
`基本設計 / 詳細設計 / コーディング / 単体テスト / 結合テスト` を既定とし、各開発工程の後に対応するレビュー工程(基本設計レビュー / 詳細設計レビュー / コードレビュー / 単体テストケースレビュー)を自動挿入する。

## Consequences

- ce2 の「見積根拠・WBS・スケジュール表」と同じ見方を再現しつつ、見積の粒度は Excel 都合から解放され実態に合う(自由小数 + 稼働率)。
- 稼働率により「専従でない担当のタスクは期間が伸びる」現実を表現できる。
- 既存の日数 ceil ベースのスケジューラ(0002/US-004)は「期間 = ceil(工数 ÷ 稼働率)」へ拡張する。稼働率 100% かつ整数人日では従来結果と一致するため後方互換は概ね保たれる。
- データモデルが複雑化(自己参照階層・工程・kind・稼働率)するため、生成ロジックは純粋関数に隔離してテスト容易性を確保する。
- 概算モードは持たない(正式一本)。将来必要になれば本 ADR を Superseded で更新する。
