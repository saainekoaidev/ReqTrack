# 0019 レビューの同時刻スケジューリングと書面レビュー

## Status

Accepted (US-044/0018 の前後直列を是正し、スケジューラを区間ギャップ充填へ刷新)

## Context

US-044 ではレビュワー側・レビュイー側のレビュー工程を別タスクとして生成したが、スケジューラが「直前終了からの直列」だったため両者が前後に直列化され、実態(レビューは双方が同席する 1 つの時間)と合わなかった。また「書面レビュー」(同席せず、依頼を出してレビュワーが単独でレビュー)を扱いたい。

## Decision

### スケジューラを区間ギャップ充填方式へ刷新 (`scheduleTasks`)

- 各要員の「埋まっている区間」を保持し、空き(ギャップ)へ**非割込み**で配置する(タスクは走り始めたら分割しない)。
- 依存: 同一 groupKey(対象)は配列順=工程順に直列(直前終了が起点)。
- 要員: 同一 resourceKey は重複不可。空きがあれば前詰め。
- **同期(syncGroup)**: 同じ syncGroup のタスクは、関係要員**全員の空きが同時に合う最早区間**へ同一区間で配置する。待っている側はその間、別タスクを進められる(前詰めされる)。

### レビュー形態 (`reviewFormat`, `reviewLinkId` を Task に追加)

- **対面(sync, 既定)**: レビュワー側+レビュイー側の 2 タスクを生成し、`reviewLinkId` で同期ペアにする → 双方の空きが合う所へ同時配置。
- **書面(doc)**: レビュワー側のみ生成(レビュイーは拘束しない=その間別作業)。
- 形態の選択は **ガント自動作成時(見積の「この見積でガントを作成」)** と **進捗管理の「スケジュール再生成」(WBS編集)** で行う(プロジェクト共通、既定=対面)。`POST /api/projects/:id/reviews { include, format }` で再展開するため、**後から書面へ変更**できる(次の再生成で反映)。

## Consequences

- 対面レビューが双方同席の 1 区間として正しく配置され、待ち時間に担当者は別作業を進められる。
- 書面レビューでレビュイーを拘束しない計画が立てられる。
- スケジューラがギャップ充填になり、空き時間を前詰め活用する(従来の単純直列より密)。順序は依然「対象順→工程順」で安定。
- 形態は現状プロジェクト共通(全レビュー一括)。レビュー単位で対面/書面を混在させる細別は将来課題。
