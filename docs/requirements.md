# 要件定義

> 各 User Story (US) は `US-NNN	<本文>` (タブ区切り) の 1 行で記述する。新しい US は冒頭に追記して逆時系列順に並べる。
> 1 US 1 Issue 1 PR を原則とする (詳細は `CLAUDE.md` の per-US merge ceremony を参照)。

US-043	マスタ管理者として, 要員に役割(プロジェクト管理者/チームリーダー/担当者)を要員管理画面で付与したい。役割により資格を決める: レビュワー資格=プロジェクト管理者・チームリーダー、作業者(ガント上のタスク担当)資格=担当者・チームリーダー(プロジェクト管理者は原則作業者にならない。ただし要員が実質1人の場合の1人対応は許容)。プロジェクト管理者はチームリーダーを兼務できる。レビュワー自動割付は最上位(プロジェクト管理者)を優先する。なぜなら誰が作業し誰がレビューできるかを役割で一貫して扱いたいから。
US-042	プロジェクト管理者として, 進捗(実績)が入っているタスクは再スケジュールで未来へ動かさないでほしい。進捗100%(完了)は計画日を固定、1〜99%(着手中)は開始を固定し工数変更で終了のみ伸縮、0%(未着手)のみ依存・要員で引き直す。前段タスク挿入・工数変更・担当変更があっても、着手済み/完了タスクは後ろへ押し戻さない(後続の未着手タスクが流れて調整される)。なぜなら既に始まった/終わった作業の実績を計画の引き直しで壊してはいけないから。
US-041	プロジェクト管理者として, 見積後(実行フェーズ)もタスクの追加/削除/変更・工数/稼働率/担当の変更を「進捗管理 > WBS編集」で行い、「スケジュール再生成」で日程に反映したい。スケジュールは(a)依存=同一対象(1.1)配下の作業を工程順に直列、(b)要員=同一要員のタスクを直列(未割当は共通プールで直列、別要員は並行)を考慮し、プロジェクト開始日を起点に全タスクを流し直す。要員の追加/変更/削減はこの再生成で反映される(要員追加=並行化で短縮、要員変更=その人の鎖が組み替え、工数変更=以降が後ろ倒し)。なぜなら計画は実行中に変わり、その影響(起点・つながり)が予測可能である必要があるから。明示的なタスク間リンク(機能をまたぐ依存)は将来課題とする。
US-040	プロジェクト管理者として, ガントチャートを「正しいガント」にしたい。(a) WBS は 機能(1.)→対象(1.1)→作業(1.1.1) の 3 階層を行に表示し、ツリーの折り畳み(展開/収納)ができる。対象は画面ID(画面名)/帳票ID(帳票名)等で、この対象の存在がインデント3(1.1.1)掲出のキーになる。AI見積も機能→対象→作業の3階層で生成する。(b) 工数列のヘッダに単位(人日)を明示する。(c) 日付表示は必ず年を含める。(d) 工数の小数部を 1 日へ切り上げる挙動は廃止し、小数人日をそのまま反映した連続時間軸(コマ/セル区切りの無い)のバーで描画する。なぜなら ce2 は「スケジュール作成処理の参考」であって Excel 風コマ表を踏襲せよという指示ではなく、小数日まで正確なガントが必要だから。
US-039	プロジェクト管理者として, AI見積は実行直後にガントへ直行せず、見積内容(タスク・工数・稼働率・見積根拠=内訳明細・担当)を確認・補正でき、「この見積でガントを作成」操作を経て初めてガントを生成したい。担当の割付もこの見積確認の段で行える(AI見積は未割当で生成→確認時に割付。レビューはPL自動、ガント後も変更可)。AIを使う流れ(要件からガントまで)も必ずこの AI見積＋確認のワンクッションを経由する。なぜなら根拠を確認し補正してから計画を確定したいから。
US-038	プロジェクト管理者として, 新規作成フローを「1.プロジェクト作成 → 2.要件登録 → 3.見積・ガント生成」の3ステップに簡素化したい。(a) プロジェクト作成後はステップ1の入力を非活性表示にし、作成中プロジェクトを破棄して最初からやり直す「やり直し」だけが可能(戻っても別プロジェクトを作り直せない)。(b) 要件登録は「要件一覧(見積明細)ファイル」1つの取込または自然文で行う(顧客要件のみ/弊社改修案込みの差はファイル内容の違いだけ。見積明細の二重アップロードは廃止)。(c) 見積前に WBS は展開しない。WBS(タスク)はステップ3で「AI見積を作る」か「手でガントを組む」かを選んだ後に生成する。(d) 各ステップは明確な「次へ」ボタンで進む。なぜなら現行は戻ると作り直せてしまい、二重アップロードや工数0のWBS残りなど分かりにくいから。
US-037	プロジェクト管理者として, 要件から「AI見積→スケジュール割付」までを一気通貫で実行し、ガントチャートを生成したい(ce2 の要件→スケジュール表に相当。出力はガント)。あわせて「ReqTrack は Web システムで VSCode 起動は不要、AI見積のみ claude CLI を使う」ことを文言で明確化する。なぜなら要件からガントまで一発で作りたいから。
US-036	プロジェクト管理者として, 見積等の AI 処理を追加課金 API ではなく現在の Claude Code 契約の使用量枠で実行したい(DGMS と同方式: `claude -p` をサブプロセス起動、API キー不要)。まず要件と参照資料から AI で見積(タスク・工数・根拠)を生成して反映する。なぜなら LLM による見積・計画を契約枠内で実現したいから。
US-035	プロジェクト管理者として, 参照資料に全文検索インデックスを作り、AI 見積時に関連資料を素早く引けるようにしたい。なぜなら既存改修の見積で関連ソース/設計を文脈として渡したいから。
US-034	プロジェクト管理者として, 参照資料スキャンは恣意的な件数で黙って打ち切らず、読めるテキストファイルを対象に取り込み、全ファイル数・読込件数・打ち切り有無を明示してほしい。見積時には登録された資料(パス保持)を参照できる前提とする。なぜなら何が取り込まれたか把握したいから。
US-033	ユーザとして, 処理中であることをヘッダに掲出し、操作はリンクではなくボタンにして実行中は非活性にしたい(スキャン等の同時実行を防ぐ)。あわせて API エラーはサーバの具体的な文言を表示したい。なぜなら状態が分かりやすく、誤操作を防ぎ、原因が掴めるから。
US-032	プロジェクト管理者として, プロジェクトが無い、またはガント(計画済みタスク)を持つプロジェクトが無い場合は進捗管理画面へ遷移できないようにしたい。なぜなら表示するガントが無い状態での進捗管理は意味がなく混乱するから。
US-031	ユーザとして, フォルダ選択枠は「参照」(サーバ側フォルダブラウザで実パス選択)とドラッグ&ドロップ、ファイルアップロード枠は「参照」とファイルのドラッグ&ドロップで選びたい。枠は 1 行の高さ(入力+参照ボタンが 1 行に収まりその行へ D&D)でスタイリッシュに。なぜなら入力が楽で見映えも良いから。
US-030	プロジェクト管理者として, 不要になったプロジェクトを一覧から削除したい(配下の要件・タスク・日報も連動削除)。なぜなら不要案件を整理したいから。削除は誤操作防止の確認を挟む。
US-029	プロジェクト管理者として, ガント初版の生成を新規作成のワークフロー(見積の次)で行い, ガント画面では計画の再生成と分かるボタン/フィードバックにしたい。なぜなら初版生成の位置と挙動を分かりやすくしたいから。
US-028	マスタ管理者として, 要員に単価(円/時)を登録し, 工数(人日×8時間)から工賃を概算したい。なぜなら工数だけでなくコストも把握したいから。
US-027	ユーザとして, 基本設定(1日の作業時間・見積の最小単位・レビュー率/下限・既定稼働率)を範囲制限のついた任意値で設定したい。なぜなら固定値ではなく案件に合わせたいから(見積の最小単位は 0.1 人日を既定とする)。
US-026	ユーザとして, 各機能から最初の画面へ戻るボタンで遷移したい(システム名クリックだけでは分かりにくい)。なぜなら迷わず行き来したいから。あわせて画面の文言から内部管理番号(US-xxx)を消す。
US-025	マスタ管理者として, 祝日をカレンダー(日本の祝日)から自動取得して登録したい。なぜなら手入力の手間を省きたいから。
US-024	プロジェクト管理者として, 見積対象をシステム構築部分(基本設計〜結合テスト)に限定したうえで, 案件を「新規」「既存」から選び, 既存の場合は登録済みの参照資料プロジェクト(改修対象の設計/ソース等の資料フォルダ)を見積の参照に含めたい。なぜなら既存改修は既存資料を参照せずに見積れないから。
US-023	ユーザとして, 新規作成系の操作をワークフロー(ステッパー)で現在位置が分かる形で進めたい。なぜなら手順の迷いをなくしたいから。
US-022	ユーザとして, 全体設定画面でタブを切り替えて基本設定・要員登録・休日登録を行いたい。なぜなら設定類を 1 か所に集約したいから。
US-021	ユーザとして, 新規作成系と進捗管理系で入口を分け, 淡い緑系統テーマ・グラフィカルフォント・ピクトグラムのアプリシェル(ヘッダ/左ペイン/右上プロジェクト選択)で操作したい。なぜなら見映えと現在位置の分かりやすさを高めたいから(特にガントは難しいと言われるため)。
US-020	プロジェクト管理者として, 新規プロジェクトを「見積から」「ガントから」のいずれかの起点で開始したい。なぜなら案件の状況に応じて入口を選びたいから。
US-019	プロジェクト管理者として, 要件一覧/見積明細のファイル(テンプレ/自由体裁)や自然文の入力から(精度は劣ってもよいので)見積とガントのタスクへ展開したい。なぜなら定型に縛られず素早く着手したいから。
US-018	プロジェクト管理者として, ガント(WBS 表)上で階層を持つタスクを一覧表上で直接 追加・削除・修正したい。なぜなら空のガントから素早く計画を組み立てたいから。
US-017	要員として, 日報で複数の WBS タスクを選んで進捗率とコメントをまとめて登録し, 一覧に蓄積・リンクから詳細確認でき, 登録内容がガントの進捗率と全体進捗へ反映され, 遅延時はコメントから要因を辿りたい。なぜなら日々の実績入力を一括で行い計画と実績を繋ぎたいから。
US-016	プロジェクト管理者として, 見積諸元と見積根拠を明記した Excel(.xlsx) を出力したい。なぜなら顧客提示用に WBS・ガント・見積根拠を 1 ファイルで共有したいから(MS Project mpp の代替)。
US-015	プロジェクト管理者として, ガントチャートを ce2 準拠の列項目(月/日+曜日・工程色・担当者・凡例)で営業日単位に表示したい。なぜなら既存のスケジュール表と同じ見方で計画を確認したいから。
US-014	プロジェクト管理者として, 開発タスク後の PL レビューを自動展開し、複数機能同時実施の効率化調整(負の工数)を表現したい。なぜなら見積合計を実態に合わせつつ削減根拠を追跡したいから。
US-013	プロジェクト管理者として, 要件から「機能→対象→作業タスク」の 3 階層 WBS と標準工程を展開し, 各タスクに工程区分と見積根拠(工数推定理由)を持たせたい。なぜならタスクの考え方を ce2 準拠で構造化し, 担当(role/assign)を継承しつつ見積根拠を残したいから。
US-012	プロジェクト管理者として, 工数を稼働率を加味して算出したい(工数は小数点第3位までの自由値, 稼働率で実作業量とカレンダー期間の関係が変わる)。なぜなら担当の専従度に応じた現実的な工数・期間を見積もりたいから。
US-011	プロジェクト管理者として, 遅延に対するリカバリプラン案を自動で作成・提示してほしい。なぜなら遅れの挽回策を素早く検討し意思決定したいから。
US-010	プロジェクト管理者として, 進捗報告をもとに遅れている要員を洗い出したい。なぜなら遅延の発生箇所と担当者を特定して対処したいから。
US-009	プロジェクト管理者として, 計画に対する遅れを検出したい。なぜなら問題が大きくなる前に早期に手を打ちたいから。
US-008	プロジェクト管理者として, 受け付けた進捗報告をガントチャート上の進捗率へ反映したい。なぜなら計画と実績を同じ基盤上で一元的に把握したいから。
US-007	要員として, 自分の担当タスクの進捗を報告したい。なぜなら実績を計画に反映し進捗管理に役立てたいから。
US-006	マスタ管理者として, 祝日を登録したい。なぜならスケジュール算出時に非稼働日を正しく考慮したいから。
US-005	マスタ管理者として, 要員を登録したい。なぜならタスクの割当や進捗管理の対象として要員を扱いたいから。
US-004	プロジェクト管理者として, 見積をもとにガントチャートの初版を作成したい。なぜなら計画の全体像を可視化し下流工程の管理の基盤にしたいから。
US-003	プロジェクト管理者として, 洗い出したタスク・工程に対して見積を作成したい。なぜなら各作業に必要な工数・期間を見積りスケジュールの根拠にしたいから。
US-002	プロジェクト管理者として, 入力された要件から作業タスクおよび工程を洗い出したい。なぜなら計画立案の単位となる作業を明確にしたいから。
US-001	プロジェクト管理者として, 顧客からヒアリングした要件または提示された要件一覧を入力として受け取りたい。なぜなら要件を起点に上流工程を開始したいから。
